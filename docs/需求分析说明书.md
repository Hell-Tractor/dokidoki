## 1. 引言

### 1.1 编写目的

本文档仅描述 Dokidoki 的**业务需求、功能需求、非功能需求、范围与验收标准**。系统架构、技术选型、数据表结构、接口协议与算法实现见《概要设计说明书》及后续设计文档。

### 1.2 项目背景

构建基于 LLM 的**角色扮演聊天应用**，聚焦：

- 虚拟角色的性格化对话与情绪陪伴；
- 角色拥有独立日程，可主动发起对话；
- 维持角色一致性，不提供专业知识问答或复杂推理。

### 1.3 产品定位

**一句话**：不是 AI 助手，而是「有性格、有日程、会主动找你的虚拟角色」。

| 维度 | 说明 |
|------|------|
| 对话目标 | 情绪价值、陪伴感、角色一致性 |
| 用户输入 | 日常闲聊、情绪倾诉、图片分享 |
| 角色输出 | 符合人设的语气与反应；可提及当前活动 |
| 目标用户 | **技术用户自部署**；个人使用为主，可少量扩展 |
| 产品气质 | 偏二次元 / Galgame；产品名 **Dokidoki** |

### 1.4 术语

| 术语 | 说明 |
|------|------|
| LLM | 大语言模型 |
| OOC | 偏离角色人设 |
| Persona | 角色人设（性格、口癖等） |
| Schedule | 角色日程（周期活动与随机事件） |
| 主动消息 | 由角色侧发起、非用户先发的消息 |
| MVP | 最小可行产品 |

## 2. 用户与场景

### 2.1 干系人

| 角色 | 诉求 |
|------|------|
| 终端用户 | 沉浸感、陪伴、角色一致、不被骚扰 |
| 部署者 | 部署简单、密钥安全 |
| 测试/运营 | MVP 阶段通过数据库维护角色配置 |

### 2.2 使用场景

- **A 日常闲聊**：低信息输入时角色以人设回应，而非 AI 助手式建议。
- **B 分享图片**：角色以人设视角 react，非图像分析报告。
- **C 多角色并行**：与多个角色独立会话，各自记录与节奏。
- **D 主动消息**：App 未打开时收到角色推送。
- **E 勿扰**：按角色单独设置勿扰时段。
- **F 连发短句**：用户连发短句，角色以多条短气泡、typing、自然延迟回复。
- **G 自然结束再聊**：角色告别后双方暂停，稍后角色再主动找来。

## 3. 范围与约束

### 3.1 交付范围

| 阶段 | 平台与能力 |
|------|------------|
| MVP | Android App；文本与图片对话；自部署后端 |
| Phase 2 | iOS、TTS、视觉定制、关系好感度等（见 §5） |
| Phase 3 | 账号体系、自动搜人设生成、STT |

### 3.2 业务约束

- 须使用**自部署后端**（LLM 密钥不可置于客户端；主动消息与日程依赖服务端）。
- 一个后端实例对应一套 LLM 配置，由部署者配置。
- App 用户通过 Server URL 连接后端，注册/登录后持有 Bearer Token。
- MVP 不提供 App 内人设/日程编辑；由测试人员维护角色数据。

### 3.3 明确不做

- Telegram Bot 等第三方通道；独立在线/离线状态（用活动状态代替）；多角色吃醋；天气/定位；全量 RAG；知识库与工具调用；长文/代码/数学问答。

### 4.8 M-07 主动消息

| 编号 | 需求描述 |
|------|----------|
| FR-07-01 | 支持触发类型：日程切换、沉默唤醒、话题重启、每日问候、情绪跟进、特殊日期 |
| FR-07-02 | 主动消息内容由角色按当前状态生成，非固定模板 |
| FR-07-03 | App 在线即时送达；离线通过系统通知 |
| FR-07-04 | 用户可配置全局日上限（默认较大，如 20 条/天） |
| FR-07-05 | 主动节奏由性格与日程驱动，非逐角色滑块 |
| FR-07-06 | 勿扰时段内不发送主动消息 |
| FR-07-07 | 每日问候在角色日程合适时段主动发出，**非**打开 App 时触发 |
| FR-07-08 | 每日问候计入全局日上限 |
| FR-07-09 | 沉默唤醒阈值由角色性格决定（粘人更短、冷淡更长；默认可配置） |

### 4.10 M-13 真实聊天（Burst Chat）

| 编号 | 需求描述 |
|------|----------|
| FR-13-01 | 用户每条短句单独显示在聊天界面 |
| FR-13-02 | 连发结束后短暂静默（默认约 2 秒）再统一触发角色回复 |
| FR-13-03 | 连发过程中新消息纳入同一轮理解 |
| FR-13-04 | 角色以 1–4 条短气泡回复，非长段落 |
| FR-13-05 | 气泡逐条显示，条间有打字感延迟 |
| FR-13-06 | 处理中显示「对方正在输入」 |
| FR-13-07 | 空闲时避免秒回，保留最短自然延迟 |
| FR-13-08 | 角色忙碌时首响延迟见 M-15 |
| FR-13-09 | 投递中用户再发消息，取消未显示内容并重新处理 |
| FR-13-10 | 支持纯 emoji 收发 |
| FR-13-11 | 上课等忙碌时回复更短 |
| FR-13-12 | Phase 2：流式按句投递、完整已读后再 typing |

### 4.11 M-14 选择性回复与话题终结

| 编号 | 需求描述 |
|------|----------|
| FR-14-01 | 角色可不回复用户消息（与性格相关） |
| FR-14-02 | 角色可主动结束话题（如「我去忙了，等下聊」） |
| FR-14-03 | 用户回复「好的」「拜拜」等告别语后，角色可不再回复（与性格相关） |
| FR-14-04 | 对话暂停一段时间后，角色可主动重新发起话题 |
| FR-14-05 | 用户发起新话题时对话恢复正常 |
| FR-14-06 | 告别类短句可识别（如：好的、拜拜、去吧、嗯嗯） |

### 4.12 M-15 忙碌时回复延迟

| 编号 | 需求描述 |
|------|----------|
| FR-15-01 | 角色忙碌时，首条回复前等待时间显著长于空闲时 |
| FR-15-02 | 等待时间受繁忙程度与性格共同影响 |
| FR-15-03 | 同一忙碌状态下等待时间具有随机性（模拟偶尔看手机） |
| FR-15-04 | 忙碌等待期间不显示「正在输入」 |
| FR-15-05 | 等待中用户再发消息，重新计算等待 |
| FR-15-06 | 角色在空闲段开始后应能更快回复 |

### 4.13 M-16 轻量记忆

| 编号 | 需求描述 |
|------|----------|
| FR-16-01 | 角色可记住用户提及的事实，并在后续对话引用 |
| FR-16-02 | 记忆可分为短期（会遗忘）与**永久**（不过期） |
| FR-16-03 | 琐事默认较快遗忘；重要事实可长期或永久保留 |
| FR-16-04 | 用户事实变化时（如喜好改变），旧记忆应被删除或覆盖 |
| FR-16-05 | 同一主题的记忆可去重，新信息覆盖旧信息 |
| FR-16-06 | 过期记忆不再影响对话 |

### 4.14 M-17 延迟已读

| 编号 | 需求描述 |
|------|----------|
| FR-17-01 | 忙碌时，用户可感知「已送达」早于「已读」 |
| FR-17-02 | 「已读」早于角色开始输入与回复 |
| FR-17-03 | 空闲时已读接近即时 |

### 4.15 M-26 用户档案

## 5. 非功能需求

### 5.1 性能与规模

| 编号 | 需求描述 |
|------|----------|
| NFR-01 | 响应速度主要受 LLM 影响，系统侧不应成为明显瓶颈 |
| NFR-02 | 单实例支撑个人及少量用户并发 |

### 5.2 安全

| 编号 | 需求描述 |
|------|----------|
| NFR-03 | LLM API Key 仅存在于服务端 |
| NFR-04 | 通信加密（HTTPS） |
| NFR-05 | 图片访问需授权 |
| NFR-06 | 上传图片有类型与大小限制 |

### 5.3 兼容性

| 编号 | 需求描述 |
|------|----------|
| NFR-07 | MVP 客户端：Android |
| NFR-08 | 客户端技术选型：Flutter（见概要设计） |
| NFR-09 | LLM 接入可配置切换厂商 |

### 5.4 异常与降级体验

| 场景 | 用户侧表现 |
|------|------------|
| 网络断开 | 发送失败可重发；已显示消息保留 |
| 后端不可用 | 提示无法连接，不崩溃 |
| LLM 失败 | 不暴露技术错误；可不回复或温和提示 |
| 连接中断 | 自动恢复；可补拉消息 |
| Token 无效 | 引导检查配置 |
