## 2. 系统架构

### 2.1 总体架构

```
┌─────────────────┐     REST / WebSocket     ┌──────────────────────────┐
│  dokidoki-app   │ ◄──────────────────────► │  dokidoki-server (Rust)  │
│  Flutter        │                          │  Axum + MySQL            │
│  Android 优先    │                          └────────────┬─────────────┘
└────────┬────────┘                                       │
         │ FCM                                             │ HTTPS
         ▼                                                 ▼
┌─────────────────┐                          ┌──────────────────────────┐
│  系统通知        │                          │  LLM API（外部）          │
└─────────────────┘                          └──────────────────────────┘
```

### 2.2 为何需要后端

| 能力 | 仅客户端 | 结论 |
|------|----------|------|
| 文本/图片对话 | API Key 暴露 | 需后端代理 |
| App 关闭时主动消息 | 不可靠 | 服务端 + 推送 |
| 日程调度 | App 被杀失效 | 服务端定时任务 |
| 多人共用部署 | 需集中存储 | 需后端 |
| 跨设备同步 | 需云端 | 需后端 |

### 2.3 部署架构

- VPS 上 **Docker Compose** 部署：`dokidoki-server` + MySQL + Caddy（HTTPS）
- 单后端实例对应一套 LLM 配置（`config.toml` / 环境变量）
- 图片 MVP 存 VPS 本地 `/data/uploads/`

## 3. 技术选型

| 层次 | 选型 |
|------|------|
| 客户端 | **Flutter**（`firebase_messaging`） |
| 后端 | **Rust** / **Axum** / **Tokio** |
| 数据库 | **MySQL** + **sqlx** |
| 定时任务 | tokio-cron-scheduler 或 `tokio::time::interval` |
| HTTP 客户端 | reqwest |
| 推送 | FCM HTTP v1（MVP Android）；APNs Phase 2 |

**后端结构（MVP）**：单 crate + 模块。顶层模块使用 `module.rs`，子模块使用 `module/sub.rs`，**不使用** `module/mod.rs`：

```
src/
  main.rs
  config.rs
  error.rs
  state.rs
  auth.rs        # 注册/登录、密码哈希、user_sessions
  persona.rs     # 人设读取、prompt 组装（子模块 persona/prompt.rs）
  schedule.rs    # 日程状态机、随机事件
  chat.rs        # 对话引擎（子模块 chat/burst.rs 等）
  proactive.rs   # 主动消息调度
  memory.rs      # 轻量记忆
  llm.rs         # LLM HTTP 客户端
  push.rs        # FCM
  scheduler.rs   # 定时任务
  api.rs         # Axum 路由聚合（子模块 api/response.rs、api/rest.rs、api/ws.rs 等）
  db.rs          # sqlx models & queries
```

**分层（MVP）**：传输层 `api` → 领域服务 `chat` `proactive` 等 → 仓储 `db` + 适配器 `llm` `push`。详见《详细设计》§2.3.1。

**项目命名**：产品 Dokidoki；仓库 `dokidoki`；后端 `dokidoki-server`；客户端 `dokidoki-app`

### 3.1 LLM 配置

全部在后端静态配置，不存数据库、不从前端传入：

```toml
[llm]
base_url = "https://api.deepseek.com/v1"
api_key  = "sk-..."          # 或环境变量 LLM_API_KEY
model    = "deepseek-chat"
vision_model = "..."         # 可选，图片对话时配置
```

一个后端实例 = 一套 LLM 配置；换模型/厂商改配置后重启即可。

## 4. 逻辑数据模型

```
users                -- username, password_hash, display_name, birthday, timezone, max_proactive_per_day
user_sessions        -- token_hash → user_id（Bearer 鉴权）
characters           -- persona_json, schedule_json (JSON)
user_character_settings  -- dnd_start, dnd_end, push_muted (Phase 2)
conversations        -- status, winding_reason, paused_at, summary, first_contact_done
messages             -- turn_id, seq_in_turn, content_type, image_path, read_at, reply_to_id
character_states     -- PK=character_id；多用户共用同一角色日程状态
user_memories        -- memory_type, memory_key, expires_at (NULL=永久)
proactive_logs
devices              -- fcm_token, platform
```

**UI 原则**：`current_mood` 仅用于 Prompt，不在 UI 展示；Phase 2 展示 `current_activity`。

### 4.1 人设与日程配置结构（示例）

**persona_json**：

```yaml
persona:
  personality_traits: ["傲娇", "口是心非"]
  speech_style:
    tone: "简短、带省略号"
    catchphrases: ["才、才不是担心你呢"]
    forbidden: ["过于正式", "像 AI 助手一样给建议"]
  reply_delay_factor: [1.0, 1.0]
  conversation_style: "回复频率和热情适中，该不回可以不回"
  conversation_behavior:
    skip_reply_tendency: "medium"
    end_topic_freely: true
    pause_on_farewell: true
  proactive:
    silence_after_hours: 8
    probability_factor: 1.0
    user_busy_reengage:
      min_delay_minutes: 30
      ramp_minutes: 90
      peak_probability: 0.6
```

**schedule_json**：

```yaml
schedule:
  timezone: "Asia/Shanghai"
  weekly_template:
    monday:
      - { start: "07:00", end: "09:00", activity: "做早餐", availability: "medium", mood: "元气", kind: "wake" }
      - { start: "08:00", end: "12:00", activity: "上课", availability: "low", mood: "困", kind: "custom" }
      - { start: "22:30", end: "07:00", activity: "睡觉", availability: "low", mood: "困倦", kind: "sleep" }
  random_events:
    probability: 0.15
    pool: ["今天社团活动取消了", "下雨，心情有点闷"]
```

时段 `kind`：`wake`（起床/晨间问候）/ `sleep` / `custom`（缺省）。服务端反序列化为 `Schedule` 结构体。

完整 JSON Schema 与 ER 图见《[详细设计说明书](./详细设计说明书.md)》§2.11、§4；Prompt 模板见《[Prompt规范](./Prompt规范.md)》。

### 4.2 MySQL 物理表（概要）

| 表 | 关键字段 |
|----|----------|
| `users` | `username` UNIQUE, `password_hash`, `display_name`, `birthday`, `timezone`, `max_proactive_per_day` |
| `user_sessions` | `user_id`, `token_hash` UNIQUE, `expires_at` |
| `characters` | `name`, `persona_json` JSON, `schedule_json` JSON |
| `user_character_settings` | `user_id`, `character_id`, `dnd_start`, `dnd_end` |
| `conversations` | `user_id`, `character_id` UNIQUE, `status`, `winding_reason`, `paused_at`, `summary`, `first_contact_done` |
| `messages` | `conversation_id`, `role`, `content`, `content_type`, `turn_id`, `seq_in_turn`, `image_path`, `read_at` |
| `character_states` | `character_id` PK, `current_activity`, `current_mood`, `random_event`, `last_proactive_at` |
| `user_memories` | `user_id`, `character_id`, `content`, `memory_type`, `memory_key`, `expires_at` |
| `proactive_logs` | 主动消息记录，用于日上限统计 |
| `devices` | `user_id`, `fcm_token`, `platform` — FCM 推送注册 |

### 4.3 配置与存储分离

| 数据 | 存储 | 维护方 |
|------|------|--------|
| LLM Key/URL/Model | 服务端配置 | 部署者 |
| 人设/日程 | MySQL | MVP：DB 直写 |
| 用户档案/设置 | MySQL | 用户 App（注册时创建，Settings 可改） |
| 鉴权 Token | MySQL `user_sessions` | 注册/登录签发 |
| 消息/图片 | MySQL + 本地文件 | 系统 |

### 4.4 图片存储（MVP）

用户上传图片存 VPS 本地 `/data/uploads/`，`messages.image_path` 存相对路径；通过后端 API 鉴权访问。后续可扩展 S3/OSS。

### 4.5 主动消息调度（概要）

1. Scheduler tick：先过滤 sleep 内 / 勿扰 / 日上限 / `winding_down`
2. 按优先级取一条：`pre_sleep` → `daily_greeting` → `re_engage` → `silence_wake` → …
3. **`pre_sleep`**：切入 sleep 前晚安；`paused_char_busy` 时可叠关心；覆盖与 sleep 首尾相接
4. **`re_engage`**：仅 `paused_char_busy`（忙完 `activity_ends_at`）或 `paused_user_busy`（人设时间→概率曲线）
5. **`silence_wake`**：仅正常结束的 `paused`
6. LLM 生成 → WS（在线）/ FCM（离线，延后）

## 6. 关键机制设计

### 6.1 Burst Chat

- 入站：静默窗口（默认 2s）合并连发 → 单次 LLM 调用
- 出站：LLM 用 `|||` 分隔 1–4 条气泡，逐条投递
- 气泡间打字延迟：约 400–1500ms/条

### 6.2 LLM 动作协议

| 动作 | 用途 |
|------|------|
| `[REPLY]` | 正常多气泡回复 |
| `[NO_REPLY]` | 不回复 |
| `[END_TOPIC]` | 告别并进入 winding_down |
| `[STORE_MEMORY]` | 写入记忆 |
| `[FORGET_MEMORY]` | 删除/覆盖记忆 |

### 6.3 会话状态机

| 状态 | 含义 |
|------|------|
| `active` / `winding_down` | 对话中 / 收束中（可再聊几句；`winding_reason` 区分 normal / char_busy / user_busy） |
| `paused` | 话题**正常**结束，不必重启 → 仅 `silence_wake` |
| `paused_char_busy` / `paused_user_busy` | 话题**异常**中断 → `re_engage` |

`winding_down` 超时默认 5 分钟（全局 config）。详见《[详细设计说明书](./详细设计说明书.md)》§9。

### 6.4 忙碌时回复延迟（M-15）

- 输入：`availability`、`persona.reply_delay_factor`、随机 jitter
- `low`：30% 偷看即回 / 45% 找空档 / 25% 等活动结束
- 性格系数：直接使用角色 `reply_delay_factor` 区间，缺省 `1.0`

### 6.5 记忆机制（M-16）

- 类型：trivial 7d / normal 30d / important 90d / **permanent** 不过期
- `memory_key` UPSERT 覆盖；`FORGET_MEMORY` 按 key 或关键词删除
- 每日清理 `expires_at < now()`

### 6.6 默认参数（实现配置）

| 参数 | 默认值 | 配置位置 |
|------|--------|----------|
| Burst 静默窗口 | 2s | config.toml |
| 摘要触发阈值 | 80 轮 | config.toml `[summary].trigger_turns` |
| 摘要后保留原文 | 40 轮 | config.toml `[summary].keep_recent_turns` |
| winding_down 超时 | 5 min | config.toml |
| 用户忙 re_engage 曲线 | 30 / 90 / 0.6 | persona `user_busy_reengage` |
| 主动日上限 | 20 条 | users 表 |
| 空闲最小延迟 | 300–800ms | config.toml |
| 沉默唤醒 | `silence_after_hours`（缺省 8h） | persona；仅 `paused` |

### 6.7 异常与降级（概要）

- LLM 失败：返回 `503`；不写入半成品角色消息；客户端提示可重试
- WS 断线：指数退避重连 + 重 subscribe；可选 REST 增量补消息
- 主动调度 LLM 失败：跳过本轮，不计日上限

详见《[详细设计说明书](./详细设计说明书.md)》§14。

## 8. 风险与对策（技术视角）

| 风险 | 对策 |
|------|------|
| LLM 超时/不可用 | 503 + 客户端重试；解析兜底 |
| WS 不稳定 | 自动重连 + REST 补拉 |
| MVP 范围大 | P0 纵向切片，P1 迭代 |
| FCM 配置复杂 | 先 WebSocket，推送跟进 |
